# ZeroController — Real-Time Body-Controlled 2D Game

> **Course context**: Artificial Neural Networks (ANN) & Deep Learning for Python/Perception (DLP)
> **Goal**: Control a 2D fighting game with the player's body, using a webcam only, with as little latency as possible.

This document is a viva-ready description of every component of the pipeline: **what** is implemented, **why** it is the right choice, and **which hyperparameters** matter.

| Stage                   | Measured latency |

## 2. The nine action classes

All are captured as **30-frame (≈1 second at 30 fps)** sequences of 33 MediaPipe landmarks.

| # | Label           | Description                                   | Mirror of  |
|---|-----------------|-----------------------------------------------|------------|
| 0 | `idle`          | Neutral standing stance                       | –          |
| 1 | `left_punch`    | Jab with the left arm                         | right_punch (auto-generated by `augment_data.py`) |
| 2 | `right_punch`   | Jab with the right arm                        | –          |
| 3 | `left_kick`     | Forward kick with the left leg                | right_kick (auto-generated) |
| 4 | `right_kick`    | Forward kick with the right leg               | –          |
| 5 | `jump`          | Short upward leap                             | –          |
| 6 | `block`         | Both arms crossed in front of chest (guard)   | – (bilaterally symmetric) |
| 7 | `forward`       | Step/lean towards the camera                  | –          |
| 8 | `backward`      | Step/lean away from the camera                | –          |

**Collection strategy**: record the **right** variant manually, then run `augment_data.py` to generate the corresponding **left** variant by flipping the x-axis and swapping left/right landmark indices. This halves the manual effort without hurting quality.

## 3. Why ST-GCN? (the central design decision)

### What the research says

Skeleton-based action recognition has converged on a clear hierarchy since 2018:

| Family | Representative models | Role today |
|---|---|---|
| RNN    | LSTM, BiLSTM | Baseline only — sequential bottleneck, weak inductive bias |
| **TCN** | Dilated 1-D conv | Fast, parallelisable, strong baseline |
| **GCN** | **ST-GCN** (Yan et al., AAAI 2018) | **Industry baseline for skeletons** |
| GCN++  | 2s-AGCN, CTR-GCN, MS-G3D (2020–22) | Higher accuracy, heavier (20–110 ms latency) |
| Hybrid | GCN + Transformer (FG-STFormer, HPI-GCN, 2023–25) | Best accuracy, needs more data |

All higher-performing models are **variants of ST-GCN**, which remains the standard baseline.

### Why ST-GCN specifically fits this project

1. **Skeleton is naturally a graph.** Joints are nodes; bones are edges. ST-GCN bakes this directly into the model's inductive bias, so the network does not have to learn "left wrist and left elbow are related" from scratch — it is given for free by the adjacency matrix. A flat LSTM/TCN over 132 features throws this structure away.
2. **Data efficient.** A strong inductive bias means fewer parameters are needed, which matters because we're collecting data **manually** (small dataset).
3. **Real-time friendly.** A lightweight 3-block ST-GCN has ~94 k parameters and runs in <10 ms on CPU — well within the 30 fps latency budget.
4. **Course-level educational value.** ST-GCN combines three core deep-learning concepts that a DLP course should cover:
   * **Graph convolutions** (spatial aggregation on non-Euclidean data)
   * **Temporal 1-D convolutions** (the TCN idea, applied to each joint)
   * **Residual connections** (the ResNet trick)
5. **Rejected alternatives**:
   * **CTR-GCN / MS-G3D** — overkill for 9 visually distinct classes; would overfit a small dataset.
   * **Transformers** — data-hungry; poor fit for a manual collection setup.
   * **Pure LSTM** — sequential inference is the slowest of all options and it lacks the graph prior.
   * **Pure TCN** — we keep it in the codebase as a comparison baseline, but it ignores the skeleton topology.

### Comparison kept in the codebase

The repo ships with all three implementations (`ActionSTGCN`, `ActionTCN`, `ActionLSTM`). Switching is a single config change (`MODEL_TYPE = "stgcn" | "tcn" | "lstm"`). This makes ablation experiments for the final report trivial — "ST-GCN +4 % vs TCN +2 % vs LSTM" is a one-flag change.

## 4. Model architecture in detail

### 4.1 Input tensor layout

After preprocessing, each sample becomes a `(C, T, V)` tensor:

* `C = 6` channels — 3 for position `(x, y, z)` + 3 for velocity `(Δx, Δy, Δz)`
* `T = 30` time steps (one sample ≈ 1 s of motion)
* `V = 33` MediaPipe BlazePose joints

This is a **channel-first** layout (the convention used by all `Conv2d` layers in PyTorch).

### 4.2 The adjacency matrix `A_hat`

Built once in `build_adjacency()`:

1. Create the symmetric binary adjacency from the 34 MediaPipe bone connections.
2. Add self-loops (`A + I`) so each joint keeps its own feature.
3. Symmetric normalization (Kipf & Welling, 2016):
   `A_hat = D^(-1/2) (A + I) D^(-1/2)`

The normalization keeps well-connected joints (like the shoulders and hips) from dominating aggregation, and it keeps activation magnitudes stable across layers.

`A_hat` is registered as a **buffer**, not a parameter — it moves with the model across devices (`.to(device)`) but is never updated by gradient descent.

### 4.4 Full model

| Layer                     | Output shape     | Notes                                       |
|---------------------------|------------------|---------------------------------------------|
| Input                     | (B, 6, 30, 33)   | positions + velocities                      |
| Input BN (over C·V = 198) | (B, 6, 30, 33)   | stabilises per-joint scales                 |
| ST-GCN block 1            | (B, 32, 30, 33)  | temporal kernel = 9, dropout = 0.3          |
| ST-GCN block 2            | (B, 64, 30, 33)  |                                             |
| ST-GCN block 3            | (B, 64, 30, 33)  |                                             |
| Global average pool (T,V) | (B, 64)          | collapses the time-joint grid to a vector   |
| Linear                    | (B, 9)           | logits                                      |
| Softmax (inference only)  | (B, 9)           | class probabilities                         |

**Total trainable parameters ≈ 94 000.** Small enough for ~500 samples without overfitting.

## 5. Preprocessing — the other half of the story

A deep model is only as good as the data it sees. All four stages below are applied **identically at training and inference time**:

### 5.1 Hip-centering
The mid-point of the two hip landmarks (index 23 & 24) is subtracted from every joint. This makes the features **invariant to where in the frame the player stands**, so the model learns the action itself, not the screen coordinates.

### 5.2 Scale normalisation
Every sample is divided by its torso length (mid-hip → mid-shoulder distance). This makes the features **invariant to body size and camera distance**, so a tall adult and a short kid look "the same size" to the network.

### 5.3 Visibility weighting
Each joint's `(x, y, z)` is multiplied by its MediaPipe visibility score. Occluded joints contribute less — their coordinates are often noisy guesses, so the model should trust them less.

### 5.4 Velocity stream
First-order temporal difference `Δposition = x[t] − x[t-1]` is appended as three extra channels. Action recognition is fundamentally about **motion**, so giving the model Δposition directly is a strong shortcut compared to forcing it to infer motion from the temporal axis alone.

### Why these flags live in the checkpoint
`train_model.py` writes every preprocessing flag into the saved checkpoint. `run_model.py` reads them back and uses those exact flags. This makes it **impossible** for training and inference to drift apart even after a config change.

## 6. Training

### 6.1 Hyperparameters (in `config.py`)

| Parameter                  | Value     | Reason |
|----------------------------|-----------|--------|
| `SEQUENCE_LENGTH`          | 30 frames | ~1 s — long enough for full motion, short enough for real time |
| `BATCH_SIZE`               | 8         | Small dataset — larger batches don't help and hurt generalisation |
| `LEARNING_RATE`            | 1 × 10⁻³  | Adam default, well-tested |
| `EPOCHS`                   | 120       | Combined with cosine schedule, reaches convergence comfortably |
| `DROPOUT`                  | 0.3       | Regularisation against overfitting with small data |
| `STGCN_CHANNELS`           | (32,64,64)| Channel growth schedule |
| `STGCN_TEMPORAL_K`         | 9         | Covers ~0.3 s around each frame; 3 blocks ⇒ receptive field > 30 frames |

### 6.2 Loss, optimiser, schedule

* **Loss**: `CrossEntropyLoss` (multi-class, includes log-softmax + NLL).
* **Optimiser**: `Adam` — fast, adaptive, works out of the box.
* **LR schedule**: `CosineAnnealingLR(T_max=EPOCHS, eta_min=1e-5)` — decays smoothly over training. This is the single hyperparameter change that most reliably improves final accuracy at no extra compute cost.

### 6.3 Train / val split
Shuffled 75 / 25 split with a fixed seed (`seed=42`). For each epoch we track both loss and accuracy and keep the checkpoint with the best validation accuracy.

### 6.4 Incremental class expansion
A clever bit in `_resume_or_build`: if a new action is added to `ACTIONS` after a model is already trained, the script **reuses the old backbone** and only adds new FC rows, using a **differential learning rate** (`0.1×` for backbone, `1×` for FC). This means you can add a new move without retraining from scratch.

### 6.5 Model merging (`merge_models.py`)
Two or more trained checkpoints can be merged: backbones are averaged, FC rows are taken per-action. Useful if one batch of data was collected with punches and another with kicks — you can train two specialists, then merge them.

## 7. Inference — turning predictions into game input

A raw classifier output per frame is noisy: a single punch can trigger 5–10 consecutive "right_punch" predictions as the buffer slides. Feeding that straight into a game would fire the punch 5–10 times. The `ActionGate` class (in `run_model.py`) fixes this with **two guards**:

### 7.1 Stability check
A label is "stable" only if it survives **N consecutive frames** (`PREDICT_STABLE_FRAMES = 3`) at softmax confidence ≥ **threshold** (`PREDICT_CONFIDENCE_THRESHOLD = 0.85`). Transient flickers are ignored.

### 7.2 One-shot trigger + cooldown
A non-idle action fires **exactly once** on the rising edge. A `cooldown` (`PREDICT_TRIGGER_COOLDOWN = 10` frames ≈ 0.33 s) blocks further triggers. Returning to `idle` re-arms the gate, so the same move can fire again after the player resets.

This turns the classifier's per-frame probability stream into **clean, one-shot game inputs** — exactly what a 2D fighting game needs.

## 8. Evaluation to include in the report

### 8.1 Quantitative
* Final **training loss / accuracy** and **validation loss / accuracy** curves
* **Confusion matrix** across 9 classes (to show where the model confuses similar motions, e.g. `forward` vs `jump`)
* **Per-class precision / recall / F1**
* Inference **latency breakdown** (MediaPipe vs preprocessing vs ST-GCN vs gate)

### 8.2 Ablation (strongly recommended for marks)
The codebase makes this one-flag easy:

| Setting                                 | What it tests |
|-----------------------------------------|---------------|
| `MODEL_TYPE = "lstm"`                   | baseline RNN |
| `MODEL_TYPE = "tcn"`                    | temporal-conv baseline |
| `MODEL_TYPE = "stgcn"`                  | recommended |
| `PREPROCESS_HIP_CENTER = False`         | importance of pose normalisation |
| `PREPROCESS_SCALE_NORM = False`         | importance of body-size invariance |
| `PREPROCESS_USE_VISIBILITY = False`     | importance of visibility weighting |
| `STGCN_USE_VELOCITY = False`            | importance of the velocity stream |

Running each and recording validation accuracy gives a convincing **feature ablation table** for the report.

## 10. How to run

```bash
# 1. Install deps (once)
pip install -r requirements.txt

# 2. Verify pose detection works
python main.py

# 3. Collect data for each of the 9 actions (keyboard: 1-9 to pick action, SPACE to record)
python collect_data.py

# 4. Mirror right-side samples into left-side samples automatically
python augment_data.py

# 5. Train the classifier (ST-GCN by default)
python train_model.py

# 6. Test live
python run_model.py
```

After every config change (e.g. adding a new action or switching `MODEL_TYPE`), rerun step 5. The incremental-expansion logic in `_resume_or_build` will reuse the old backbone when possible.
